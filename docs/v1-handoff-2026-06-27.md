## Shipped State

The main portal Operations Home now reads live work-engine data through the
`/work/api/*` broker.

It shows the operator:

- work due today
- overdue tasks
- waiting tasks and due follow-ups
- active bundles
- notification counts from the work engine

The work engine supports waiting and follow-up state:

- tasks can be marked `waiting`
- tasks can store `waitingFor`
- tasks can store `followUpAt`
- due waiting tasks generate follow-up reminders from `GET /api/notifications`
- reminder generation is idempotent for `taskId + followUpAt`, including
  reminders the operator already dismissed

The work-engine dashboard has follow-up actions:

- `Response received` moves a waiting task back to `todo`
- `Follow-up sent` keeps the task waiting and sets the next follow-up date
- both actions append an event line to the task note

Task completion now records server-side completion metadata:

- `completedBy`
- `completedAt`

The portable export includes waiting state, follow-up fields, reminders, and
completion metadata.

## Known Gaps

The main portal shows live cards, but it doesn't yet let the operator finish
the daily work from one smooth screen. The work-engine dashboard has the new
follow-up buttons, but the portal should either expose those actions directly or
open the right workflow detail view.

The follow-up history is still a simple note append. That's enough for V1
traceability, but it isn't a structured communication log.

Notification creation happens when the operator reads notifications. That keeps
V1 simple, but a later version may need a scheduled job if we want proactive
email, Slack, or calendar reminders.

`completedBy` is only present when the request includes `x-user-id`. Existing
completed tasks won't have completion metadata unless we backfill them.

The export contains the new fields, but import and restore rehearsal are still
missing. Treat export as migration-ready data, not as a proven restore process.
